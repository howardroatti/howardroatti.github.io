🧯 Data Science: A Brief History of Things We Forgot to Monitor

Season 1 — Episode 14

🎙️ Original Episode

The model went stale.
The pipeline stopped running.
The dashboard didn’t update.

Nobody noticed for 3 weeks.

Why?

  • No alerting
  • No ownership
  • No documentation

“Monitoring later” is how ghost models happen.

Make monitoring a first-class citizen. Not an afterthought.

🔍 Let’s Go Deeper…

Monitoring is often seen as a final step — a box to check after deployment.

But in reality, it’s what keeps data systems alive. Without it, you’re building sandcastles.

🧯 What Needs to Be Monitored?

  • Data pipelines: Freshness, volume, schema, success/failure
  • Models: Accuracy, drift, input/output distributions
  • Dashboards: Update frequency, KPI anomalies, broken links
  • APIs: Latency, error rates, throughput

If it can break silently, it must be monitored.

💻 Hands-on – Simulated Monitoring Setup

import pandas as pd
import numpy as np

# Simulate model output drift
np.random.seed(42)
preds_day1 = np.random.normal(loc=0.5, scale=0.1, size=1000)
preds_day30 = np.random.normal(loc=0.65, scale=0.1, size=1000)

df = pd.DataFrame({{
    "day1": preds_day1,
    "day30": preds_day30
}})

# Compare distributions
drift_detected = abs(df["day1"].mean() - df["day30"].mean()) > 0.05
print("⚠ Drift detected?" , drift_detected)

✅ Lessons Learned

  • Monitoring should be part of the design, not just the ops phase
  • If no one owns it, it won’t be fixed
  • Alerts are only useful if they’re noticed and understood

⚠️ Common Mistakes

  • Monitoring only failures, not changes
  • Not simulating outages or breaks before going live
  • Creating alerts no one reads
