🖥️ Data Science: “It Works on My Machine”

Season 1 — Episode 4

🎙️ Original Episode

Data science is a field where you can spend hours tuning a model, writing perfect code, and getting amazing results — only for everything to break on another computer.

  • Model has 99% accuracy in training but fails in production.
  • Pipelines break mysteriously overnight.
  • "Simple" analyses take three days due to messy data.
  • Stack Overflow solutions are copy-pasted with hope.

Why We Stick With It

  • Scripts that run without errors feel like magic.
  • Clear data visualizations are oddly satisfying.
  • Correct model predictions = digital joy.
  • Automation = power.

So if a data scientist says “It works on my machine,” give them a hug. They’re struggling.

🔍 Let’s Go Deeper…

This phrase usually means: environment mismatch, missing files, bad paths, or no version control. But we can fix that.

🧱 Common Causes

  • Different library versions (e.g., pandas 1.4.2 vs 2.0.0)
  • Missing local files or hardcoded paths
  • No random seed set
  • Unserialized models or non-versioned data

💻 Hands-on – Making Your ML Workflow Reproducible

import pandas as pd
import numpy as np
from sklearn.ensemble import RandomForestClassifier
import joblib
import os

np.random.seed(42)
X = np.random.rand(100, 5)
y = np.random.randint(0, 2, size=100)

model = RandomForestClassifier(random_state=42)
model.fit(X, y)

os.makedirs("models", exist_ok=True)
joblib.dump(model, "models/rf_model.pkl")

df = pd.DataFrame(X, columns=[f"feature_{i}" for i in range(5)])
df.to_csv("data/sample.csv", index=False)

🛠️ Reproducibility Checklist

  • [x] Use requirements.txt or pyproject.toml
  • [x] Set random seeds
  • [x] Save models and data artifacts
  • [x] Avoid hardcoded paths
  • [x] Document your pipeline
  • [x] Use version control (Git)

✅ Lessons Learned

  • Reproducibility = collaboration and deployment readiness.
  • Docker, virtualenv, and notebooks help a lot.
  • Tracking artifacts and seeds helps debugging.

⚠️ Common Mistakes

  • Not logging environment or versions.
  • Skipping .gitignore and polluting repos.
  • Using notebook state instead of reproducible scripts.

🎯 Practice Challenge

  • Pick a recent model.
  • Export model, data, and notebook.
  • Create requirements.txt.
  • Share with someone — can they run it?

📎 Tools & Resources

🧠 Final Thought

“It works on my machine” is just the beginning. Let’s build things that work everywhere, not just in our notebooks.
